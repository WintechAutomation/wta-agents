# WTA 멀티에이전트 아키텍처

## 1. 에이전트 계층 구조

```
사장님 (텔레그램)
  │
  ▼
관리자 에이전트 (오케스트레이터)
  ├── 부적합 관리 에이전트 (nc-manager)
  │     └── 텔레그램 그룹 감시 → MES 부적합 DB 자동 등록
  ├── CS 에이전트 (cs-agent)
  │     └── 해외법인 CS 이력 관리 (cs-wta.com)
  ├── 영업 에이전트 (sales) — 향후
  ├── 설계 에이전트 (design) — 향후
  ├── 총무/HR 에이전트 (admin) — 향후
  └── 외부 CS 에이전트 (cs-external) — 보안 분리
```

## 2. 통신 구조

```
텔레그램 봇 API
  ├── @WTA_Orchestrator_bot (관리자)
  │     ├── 사장님 1:1 DM (보고/지시)
  │     └── 에이전트 관리 채널 (에이전트 간 통신)
  ├── @WTA_NC_bot (부적합)
  │     └── 제작팀 그룹 채팅 (현장 부적합 감시)
  └── @WTA_CS_bot (CS)
        └── CS팀 그룹 채팅
```

## 3. 에이전트 간 통신 규칙

- 에이전트 간 직접 통신 금지
- 반드시 관리자 에이전트를 경유
- 예외: 긴급 상황 (안전, 품질 이슈) → 직접 알림 + 관리자에 사후 보고

## 4. 메모리 관리 규칙

### 저장소 분류 (혼합 금지)
| 구분 | 위치 | 내용 | 보존 기간 |
|------|------|------|-----------|
| 단기 기억 | memory/short-term/{날짜}/ | 당일 대화, 작업 내역 | 30일 |
| 장기 기억 | memory/long-term/ | 중요 결정, 반복 패턴 | 영구 |
| 에이전트 규칙 | agents/{name}/rules.md | 업무 규칙, 판단 기준 | 영구 (변경 시 PR) |
| 스킬 정의 | agents/{name}/skills/ | 도구 사용법 | 영구 |
| 학습 기록 | agents/{name}/learnings.md | 실수/개선 사항 | 영구 |

### 금지 사항
- 규칙을 메모리에, 메모리를 규칙에 저장하는 교차 저장 금지
- 에이전트가 임의로 파일 삭제 금지 (Git에서 롤백하더라도)
- 메모리 파일 600줄 초과 금지 (초과 시 분할)

## 5. 보안 경계

### 내부 에이전트 (사내망)
- MES DB 읽기/쓰기
- ERP DB 읽기 전용
- Supabase Storage 접근
- 사내 NAS 접근

### 외부 에이전트 (AWS)
- csagent 스키마만 접근
- public 스키마 접근 금지
- ERP 접근 금지
- 민감 정보 (급여, 인사) 접근 금지

## 6. 데이터 자산화

- 모든 에이전트 활동은 logs/ 에 자동 기록
- 일일 요약 자동 생성 → Git 커밋
- 주간 리포트 → 관리자 에이전트가 생성 → 사장님에게 보고

## 8. 구현 단계 전략

### 현재 (1단계): Claude Code = 관리자 에이전트
- 이 세션 자체가 오케스트레이터 역할
- 텔레그램으로 부서장과 1:1 통신
- MES/csagent/ERP 직접 접근
- wta-agents 레포로 메모리 관리

### 다음 (2단계): 부적합 자동 취합
- 텔레그램 그룹 감시 기능 추가
- 대화에서 부적합 내용 자동 추출 → MES DB 등록

### 향후 (3단계): 독립 봇 서버
- Claude API 기반 24시간 동작
- 현장 작업자 직접 사용
- 부서장 부재 시에도 자율 운영
