알려진 버그 (2026년 4월 기준, 현재 진행 중)

Ollama Issue #15260: think=false 설정 시 format 파라미터(JSON 스키마 구조화 출력)가 완전히 무시됨

  • 현상: think=false 지정 시 모델이 JSON 대신 일반 텍스트 출력
  • 원인: Ollama가 thinking 종료 토큰(<|/think|>) 발생 시점에 format 마스킹을 적용하는데, think=false이면 해당 토큰이 발생하지 않아 마스킹 미작동
  • 동일 버그: qwen3.5 시리즈에서도 동일 패턴 확인 (Issue #14645)
  • 임시 해결책: think 파라미터 아예 생략 → format이 정상 작동하나, thinking mode 기본 활성화로 3~4초 추가 지연 발생

llama.cpp Issue #21384: { 또는 } 문자가 string 값 내에 포함될 경우 tool call 배열 파라미터가 JSON 문자열로 직렬화되는 버그

  • LightRAG의 structured delimiter(<|entity|>, <|relation|> 등)와 함께 사용 시 파싱 실패 위험 존재
  • 결론: Ollama에서 gemma4의 구조화 출력은 현재 안정적이지 않음

6. 결론: WTA GraphRAG 자동 추출 적합/부적합 판정

gemma3:12b (현재 사용 모델)

판정: 부적합

  • structured delimiter(<|entity|>, <|relation|>, <|SEP|> 등) 생성 실패 확인
  • 12B 규모에서는 LightRAG 형식의 복잡한 구조적 출력을 안정적으로 생성하지 못함
  • instruction following 능력이 GraphRAG 추출에 요구되는 수준에 미달

gemma4:12b (현재 이슈 모델)

판정: 부적합

  • Gemma 4 라인업에 12b 모델 없음 (e2b/e4b/26b/31b만 존재)
  • gemma3:12b를 gemma4로 잘못 인식했을 가능성
  • 만약 gemma4:e4b(~4B 활성)를 의미한다면, 활성 파라미터 부족으로 GraphRAG 추출 부적합

gemma4:26b / gemma4:31b

판정: 조건부 적합 (버그 해소 후 재평가 필요)

  • 공식적으로 structured JSON output + function calling 지원
  • 그러나 Ollama 환경에서 think=false + format 조합 시 구조화 출력 무력화 버그 현존 (2026-04-05 미해결)
  • LightRAG delimiter 방식(JSON 스키마가 아닌 커스텀 구분자)은 가이드드 디코딩 없이 프롬프트 의존적 → 불안정
  • 임시 해결: think 파라미터 생략 + 프롬프트 엔지니어링으로 delimiter 강제 유도 시도 가능하나 불안정

7. 대안 권장 모델 Top 3

1위. qwen2.5-coder:32b — 즉시 사용 권장

ollama pull qwen2.5-coder:32b

선정 이유:

  • JSON 및 구조화 코드 출력에 특화 훈련됨
  • LightRAG delimiter 형식과 가장 높은 호환성
  • Ollama에서 구조화 출력 안정성 검증됨
  • 32B Dense 모델로 제조업 기술 문서 이해도 우수
  • WTA 현재 서버 환경(182.224.6.147:11434)에서 q4_K_M 기준 ~20GB VRAM으로 운용 가능
  • 현재 WTA GraphRAG POC 사용 중이며 기존 검증 결과 존재

Ollama 설치:

ollama pull qwen2.5-coder:32b

2위. qwen3:30b-a3b — 속도/품질 균형 최적

ollama pull qwen3:30b-a3b

선정 이유:

  • Qwen3-30B MoE 모델: 활성 파라미터 3B로 토큰 생성 속도 20~40% 빠름
  • IFEval 명령어 준수율 업계 최상위권 (~88+)
  • structured output 안정성 높음
  • 256K 컨텍스트 창 — 장문 제조업 매뉴얼 처리 적합
  • qwen2.5-coder:32b 대비 동일 VRAM에서 처리 속도 우위
  • LightRAG 공식 권장 오픈소스 모델로 언급됨

주의: Ollama 태그명 qwen3:30b-a3b 또는 qwen3:30b 확인 후 설치

3위. gemma4:31b-it-q4_K_M — Ollama 버그 해소 후 강력 후보

ollama pull gemma4:31b-it-q4_K_M

선정 이유:

  • 오픈소스 모델 세계 3위 (2026년 4월 기준)
  • 256K 컨텍스트, 네이티브 function calling + JSON structured output
  • 31B Dense 아키텍처로 파인튜닝 베이스로도 최적
  • τ2-bench 에이전틱 도구 사용 86.4% — 복잡한 추출 파이프라인에 적합

현재 제약:

  • Ollama think=false + format 조합 버그 미해결 (2026-04-05)
  • think 파라미터 생략 + 명시적 JSON 프롬프트 방식으로 우회 가능하나 지연 증가
  • 버그 해소 이후 즉시 전환 고려
