0. 설계 원칙

  1. 도메인 엔티티(db-manager) + 시스템 엔티티(crafter)를 하나의 그래프에 통합.
  2. FK 실존 관계와 논리적 관계를 구분 표기.
  3. PoC 단계에서는 라벨 수 최소화(15개 내외), 속성은 핵심만.
  4. Naming: 라벨은 PascalCase 단수형, 관계는 SNAKE_CASE 동사.

3. 핵심 질의 경로 (PoC 검증용)

Q1. "Korloy 포장기 #6 혼입검사 밝기 비대칭 이슈와 유사한 과거 사례는?"

(i:Issue)-[:SIMILAR_TO*1..2]-(j:Issue)-[:RAISED_ON]->(u:EquipmentUnit)
-[:INSTALLED_AT]->(c:Customer {company_name_ko:'한국야금'})
WHERE i.title CONTAINS '밝기 비대칭'

Q2. "이 이슈 담당자와 과거 유사 건 해결자는?"

(i:Issue)-[:HANDLED_BY]->(p:Person),
(j:Issue)-[:SIMILAR_TO]-(i)-[:HANDLED_BY]->(p2:Person)

Q3. "Korloy 고객 전체 CS 이력 + 관련 매뉴얼?"

(c:Customer)<-[:INSTALLED_AT]-(u:EquipmentUnit)
<-[:RAISED_ON]-(i:Issue)<-[:DOCUMENTS]-(d:Document)

Q4. "조명(DOMELIGHT100) 관련 과거 이슈 전부"

(p:Part {name:'DOMELIGHT100'})<-[:MENTIONS]-(d:Document)-[:DOCUMENTS]->(i:Issue)

Q5. "이 장비 기종에 설치된 광학계 구성 부품은?"

(m:EquipmentModel)-[:HAS_BOM]->(p:Part {category:'광학계'})

6. 조율된 사항 (crafter ↔ db-manager 병합 결과)

항목crafter 제안db-manager 제안통합 결정
도메인 엔티티Entity(type, name) 추상화실제 테이블별 엔티티db-manager 안 채택 (구체 라벨)
Document 통합단일 Document + source 필드분리 (ManualDoc/WTADoc/CSEmbedding)crafter 안 채택 (:Document 상위 라벨 + sub 라벨)
Issue 통합Issue 단일 라벨ProductionIssue/CSHistory/SetupIssue 분리crafter 안 채택 (Issue + source_type 속성)
ManualDoc 분류source=manual 단일부품매뉴얼/WTA기술문서 구분doc_type 속성 추가 (parts_manual/wta_tech)
Channel-Agent 관계OWNS + ROUTED_TO 병존—ROUTED_TO 단일 통합 (의미 중복 제거)
shape(C/W형) 위치Product 속성DB 컬럼 없음PoC 1단계 제외, 2단계 문서추출 방식 재검토
